# 系统总说明怎么写

这份文档是写给团队成员和 AI 一起用的。

它的目标不是“做一份好看的介绍”，而是让一个刚接手项目的人，或者一个刚开始协作的 AI，能在很短时间内搞清楚：这个系统是干什么的、主要由哪些部分组成、哪些地方不能乱动、遇到问题应该先看哪里。

如果项目会长期维护，这份文档一定要有，而且要持续更新。

---

## 一、为什么要写

项目小时，很多事情可以靠人脑记住：

- 登录怎么走
- 哪几个模块是主流程
- 哪些地方是从 base 改出来的
- 某个看起来重复的结构，为什么不能合并

但项目一大，这些信息会越来越分散：

- 一部分在代码里
- 一部分在历史聊天里
- 一部分在老同事脑子里
- 一部分在很久以前写过、但没人再看的文档里

这时候如果没有系统总说明，AI 和新同事最容易出现 4 个问题：

1. **恢复上下文太慢**
   不知道先看哪里，只能从代码里一层层猜。

2. **容易乱改**
   只看当前需求，就去改表面代码，结果动到了主流程、权限、公共结构。

3. **不知道哪里已经偏离 base**
   看到当前实现和 base 不一样，以为是“写错了”，结果把项目已有定制能力改没了。

4. **重复造轮子**
   不知道系统里已经有类似能力，又新造一套。

所以，系统总说明本质上是在解决两件事：

- 让人和 AI 能快速恢复项目理解
- 让后续修改先建立判断，再动代码

---

## 二、这份文档应该写哪些内容

系统总说明不是把所有细节都塞进去，而是回答下面这些最关键的问题。

### 1. 这个系统是干什么的

先用几句话说清楚：

- 系统服务谁
- 主要业务目标是什么
- 包含哪些端
- 日常最核心的使用场景是什么

例如：

- 后台端给运营和管理员使用
- 服务端负责接口、权限、数据处理
- Uni 端给终端用户使用
- 主流程是“登录 → 浏览 → 下单 → 支付 → 订单处理”

这部分不用写太长，但一定要让第一次接手的人看得懂。

### 2. 系统由哪些大模块组成

写清楚项目的主要模块，以及每个模块负责什么。

建议至少写：

- 模块名
- 主要职责
- 归属哪一端
- 和其他模块的依赖关系
- 是否属于核心模块

例如：

- 用户与权限
- 商品
- 订单
- 支付
- 消息通知
- 文件上传
- 系统配置

### 3. 哪些是核心模块

核心模块一定要单独标出来。

因为长期维护时，真正容易出事故的，不是普通 CRUD，而是这些地方：

- 登录、鉴权、权限
- 支付、下单、结算
- 主数据
- 多端共用的数据结构
- 公共组件、公共方法、公共拦截器
- 影响全局行为的配置

只要一个人或 AI 不知道“这里是核心模块”，就很容易把局部优化做成全局事故。

### 4. 系统主流程怎么走

把最重要的业务流程写出来。

不是要画很复杂的流程图，而是用人能快速扫懂的方式，写清楚：

- 从哪里进入
- 经过哪些关键模块
- 关键状态怎么变化
- 哪些节点会影响其他端或其他角色

比如：

- 用户登录流程
- 下单流程
- 审批流程
- 数据同步流程
- 权限生效流程

### 5. 当前项目和 base 的关系

如果这个项目是从 base 演化出来的，这一段非常重要。

要说明：

- 当前项目基于哪个 base 起步
- 哪些地方基本保持一致
- 哪些地方已经明显偏离
- 遇到冲突时，以什么为准

这里不是写全部差异明细，而是给读者一个判断原则：

- **当前项目已有实现优先**
- **base 主要用于参考未定制部分**
- **不要看到和 base 不一样就机械回退**

### 6. 关键目录和关键文档入口

AI 和新人最需要的是“先看哪里”。

所以要列出：

- 关键代码目录
- 关键模块入口
- 关键配置文件
- 必读文档
- 修改核心模块前要先看的文档

例如：

- `docs/README.md`
- `docs/00-system/`
- `docs/checklists/`
- 某模块的 `01-modules/模块名/`
- 某端的 skill 或开发规范

### 7. 修改前的基本规则

系统总说明里最好直接写清楚一些判断规则，让 AI 和团队成员在开始前先停一下。

例如：

- 复杂功能先看系统说明，再看模块文档
- 涉及核心模块时，不要直接改代码
- 涉及公共结构、主流程、权限、数据模型时，先确认影响范围
- 如果当前项目和 base 不一致，优先核对当前项目为什么这样实现

这类规则能明显减少“没看背景就开改”的问题。

## 三、哪些内容最重要

如果时间有限，优先把下面几项写好。

### 最重要的 1：系统目标和端划分

让读者先知道系统整体是怎么分的，不然很容易一开始就把后台、服务端、移动端混着理解。

### 最重要的 2：核心模块列表

这是避免乱改最直接的一部分。

最好明确写出：

- 哪些模块属于核心模块
- 改它们前必须先看什么
- 哪些改动需要额外确认

### 最重要的 3：主流程

项目越大，越不能只靠代码文件名猜流程。

如果主流程没写清楚，AI 很容易只改到局部，看不到整个链路。

### 最重要的 4：当前项目与 base 的判断原则

很多误改都不是因为不会写代码，而是因为不知道“该参考 base，还是该尊重当前项目”。

所以这一条一定要直接写出来。

### 最重要的 5：文档入口

好的系统总说明，不是把所有信息都写全，而是让读者知道下一步去哪里看。

## 四、什么不要写得太虚

下面这些写法，看起来像在写文档，其实对长期维护帮助很小。

### 1. 不要只写口号

比如：

- 系统采用先进架构
- 模块解耦良好
- 代码规范统一
- 具备高可扩展性

这些话如果没有对应的落点，对 AI 和团队成员几乎没有帮助。

更好的写法是直接写：

- 权限统一收口在哪个模块
- 多端共用字段定义在哪
- 哪个目录下的组件属于公共组件
- 哪个流程一改就会影响订单和支付

### 2. 不要把实现细节写成流水账

系统总说明不是逐文件解释代码。

它应该告诉读者：

- 先看什么
- 哪些地方重要
- 哪些地方风险高
- 当前项目哪里和 base 不一样

太细的实现细节，应该放到模块文档里。

### 3. 不要只写“理想结构”，不写“当前真实情况”

很多项目文档的问题是：写的是设计时想要的系统，不是现在真实跑着的系统。

如果当前项目已经偏离 base，或者有历史兼容结构，就应该写出来。

不然 AI 会按“理想模型”改，结果把真实系统改坏。

### 4. 不要长期不更新

系统总说明不是一次性交付物。

只要出现下面这些变化，就该更新：

- 新增核心模块
- 主流程变化
- 权限边界变化
- 外部依赖变化
- 项目和 base 的差异明显变多
- 关键目录或入口迁移

## 五、推荐结构

下面给一个适合长期维护项目的推荐结构。团队可以直接照着写。

```md
# 系统总说明

## 1. 系统一句话说明
- 这个系统服务谁
- 主要解决什么问题
- 包含哪些端

## 2. 系统范围
- 本项目包含什么
- 不包含什么
- 与其他系统怎么协作

## 3. 端与模块划分
### 3.1 服务端
### 3.2 后台端
### 3.3 用户端

## 4. 核心模块
- 模块名
- 作用
- 为什么是核心模块
- 修改前必须先看什么

## 5. 关键业务流程
- 登录流程
- 主业务流程
- 审批/同步/结算等高风险流程

## 6. 当前项目与 base 的关系
- 基于哪个 base
- 哪些部分保持一致
- 哪些部分已经偏离
- 发生冲突时以什么为准

## 7. 关键目录与文档入口
- 代码目录
- 文档目录
- checklist
- 模块文档入口

## 8. 修改规则
- 哪些改动不能直接下手
- 哪些任务要先查文档
- 哪些情况必须先确认影响范围

## 9. 更新规则
- 什么情况下必须更新这份文档
- 谁改了就顺手更新什么
```

## 六、推荐写法示例

可以把“核心模块”写成这种简单格式：

```md
## 核心模块

### 1. 用户与权限
- 作用：控制登录、角色、菜单权限、接口权限
- 为什么重要：会影响后台、服务端、部分用户端能力
- 修改前先看：系统登录流程、权限相关模块文档、项目与 base 差异记录
- 注意：不要只改页面判断，要一起确认接口鉴权和角色配置
```

把“项目与 base 的关系”写成这种格式：

```md
## 当前项目与 base 的关系

- 当前项目起步于 xxx base 版本。
- 后台基础 CRUD 风格大体保持一致。
- 权限、登录态、菜单结构、订单流程已经做过定制。
- 处理相关任务时，优先以当前项目实现为准；base 只用于补齐未定制部分的参考。
```

## 七、给团队和 AI 的落地建议

最后给一个最实用的原则：

### 如果要让 AI 在大项目里更稳，系统总说明至少要做到这 3 点

1. **让 AI 先知道全局结构，而不是直接看到一个需求就开改**
2. **让 AI 知道哪些模块不能凭直觉改**
3. **让 AI 知道当前项目和 base 的真实关系**

这样做之后，AI 在处理长期维护任务时，才更像一个先恢复项目理解、再动手的协作者，而不是一个只根据临时上下文输出代码的工具。

# 项目与 base 差异怎么记录

这份文档是给团队成员和 AI 用来判断“当前项目到底改到什么程度了”的。

很多长期维护项目，最容易出问题的地方，不是不会开发，而是**不知道当前项目已经和 base 偏离了多少**。

一旦这个信息缺失，就很容易出现两种错误：

- 该参考当前项目的时候，误按 base 去改
- 该参考 base 的时候，又重复造一套现成能力

所以，差异记录不是可有可无的补充文档，而是长期维护时非常关键的“判断依据”。

---

## 一、为什么要记录

### 1. 让 AI 能快速恢复上下文

AI 每次开始任务时，看到的往往只是当前需求和少量代码。

如果没有差异记录，AI 很难快速知道：

- 哪些结构还是 base 原样
- 哪些地方已经做过项目定制
- 当前项目为什么不能直接照 base 改

结果就是：

- 错把定制结构当成历史脏代码
- 错把项目特性改回模板默认行为
- 不知道哪里该复用 current project，哪里该参考 base

### 2. 让团队成员少踩坑

团队里的人换了一批之后，很多“当初为什么这么做”的背景很容易丢。

差异记录能帮后来的人快速知道：

- 哪些改动是业务需要
- 哪些改动是技术原因
- 哪些改动会影响其他模块

### 3. 避免机械回退到 base

base 是参考基线，不是所有项目都要永远保持完全一致。

如果当前项目已经因为业务、流程、权限、部署环境做了定制，就应该明确写出来。

这样后续维护的人和 AI 才不会看到不同就想“修正”回 base。

---

## 二、哪些差异必须记

不是所有差异都值得记。

像下面这些很细的小差异，一般不需要专门记录：

- 一个按钮文案不一样
- 某个页面多了一个普通筛选项
- 某个列表默认排序不同

真正**必须记**的，是那些会影响理解、扩展、迁移、排查和决策的差异。

### 1. 核心流程差异

例如：

- 登录流程和 base 不同
- 权限生效方式不同
- 下单、支付、审批、同步流程不同
- 状态流转规则和 base 不同

这类差异一定要记，因为它直接决定能不能安全修改。

### 2. 核心模块结构差异

例如：

- 公共能力放置位置变了
- 模块拆分方式和 base 不同
- 核心实体、服务、接口组织方式改了
- 菜单、路由、权限、配置结构做了重组

这类差异如果不记，后来的人很容易按 base 的目录和模式去找，结果越找越乱。

### 3. 数据结构差异

例如：

- 表结构和字段定义与 base 不同
- 状态字段扩展了业务状态
- 核心主数据模型改了
- 某些字段已经不再兼容 base 默认逻辑

涉及数据库和数据结构的差异，通常都是高影响项，必须记。

### 4. 多端协作差异

例如：

- 后端、后台、用户端之间的接口契约和 base 不同
- 某个流程从“后台发起”改成了“用户端发起”
- 角色权限划分和 base 不同

这种差异不记，跨端任务特别容易误改。

### 5. 外部依赖或部署方式差异

例如：

- 接了 base 默认没有的第三方服务
- 文件上传、消息通知、支付方式被替换
- 配置加载方式和 base 不同
- 部署拓扑与 base 明显不同

这种差异会影响排查和上线，也要记。

---

## 三、什么时候更新

差异记录最怕的不是没有，而是“曾经写过，但早就过期了”。

建议在下面几种情况下更新。

### 1. 新增明显偏离 base 的实现时

比如：

- 新增一套和 base 不同的权限逻辑
- 重写了某个核心模块结构
- 新增了 base 没有的主流程

### 2. 修改了已有差异时

如果一个模块本来就已经偏离 base，这次又继续调整了，就应该顺手更新。

### 3. 做高影响任务完成后

例如：

- 登录
- 权限
- 订单
- 支付
- 配置中心
- 公共组件体系
- 公共接口结构

这类任务结束后，最好检查一下差异记录要不要同步。

### 4. base 升级判断前后

如果团队准备参考新版本 base，或者拿 base 某段实现做迁移参考，最好先看并更新差异记录。

否则很容易低估项目已经偏离的程度。

---

## 四、记录到什么程度最合适

差异记录不要写得太空，也不要写成逐行 diff。

最合适的程度是：

- 能让读者快速知道**差在哪**
- 能让读者知道**为什么差**
- 能让读者知道**后续应该以谁为准**
- 必要时能知道**要去哪里继续看细节**

### 推荐每条差异至少写 4 件事

1. **差异点是什么**
2. **为什么会这样**
3. **后续改动时以什么为准**
4. **相关代码或文档入口在哪里**

这样最实用。

### 不建议写成两种极端

#### 极端 1：只写一句“这里和 base 不一样”

问题：没有说明哪里不一样、为什么不一样，几乎没法用。

#### 极端 2：把所有代码差异逐条贴出来

问题：维护成本太高，很快过期，反而没人看。

差异记录更像“导航”和“判断说明”，不是替代代码 diff 工具。

---

## 五、推荐记录方式

建议把差异按模块或主题来记，而不是按时间流水账来记。

比如按下面几类：

- 用户与权限
- 菜单与路由
- 订单流程
- 数据模型
- 公共组件
- 外部依赖
- 部署与配置

这样后面查找更快，AI 也更容易先按模块恢复理解。

## 六、简单模板

下面给一个团队可以直接照用的简单模板。

```md
# 项目与 base 差异记录

## 1. 使用说明
- 当前项目基于哪个 base 起步
- 这份文档的作用是什么
- 发生冲突时的基本判断原则

## 2. 总体判断
- 哪些部分基本保持 base 风格
- 哪些部分已经明显偏离
- 当前维护时优先参考哪里

## 3. 具体差异

### 3.1 用户与权限
- 差异点：
- 为什么这样做：
- 后续修改以什么为准：
- 相关入口：

### 3.2 菜单与路由
- 差异点：
- 为什么这样做：
- 后续修改以什么为准：
- 相关入口：

### 3.3 订单流程
- 差异点：
- 为什么这样做：
- 后续修改以什么为准：
- 相关入口：

## 4. 不要机械回退的部分
- 哪些地方看到和 base 不一样也不要直接改回去
- 原因是什么

## 5. 需要继续补充的差异
- 目前已知但还没整理完整的部分
```

## 七、推荐写法示例

```md
## 用户与权限
- 差异点：当前项目不是直接沿用 base 默认角色菜单关系，而是增加了按业务线拆分的角色层。
- 为什么这样做：不同业务线权限边界不同，不能只靠 base 的默认管理角色处理。
- 后续修改以什么为准：涉及角色、菜单、接口权限时，优先以当前项目实现为准；不要直接按 base 角色结构回退。
- 相关入口：系统总说明、权限模块文档、相关代码目录。
```

再比如：

```md
## 订单流程
- 差异点：当前项目订单状态比 base 多出“待确认”“人工复核”两个中间状态。
- 为什么这样做：当前项目有人工审核环节，不能直接走 base 的默认自动流转。
- 后续修改以什么为准：涉及状态判断、列表筛选、支付后流程、后台操作按钮时，先按当前状态流转核对，不要直接套 base 逻辑。
- 相关入口：系统总说明、订单模块技术设计文档、订单状态说明。
```

## 八、团队和 AI 使用这份文档的建议

### 做复杂任务前，先看这份文档

尤其是下面几类任务：

- 核心模块修改
- 高风险流程修改
- 公共结构调整
- 想参考 base 现成实现时
- 怀疑当前项目已经做过深度定制时

### 做完高影响改动后，顺手更新

最理想的做法不是等“文档整理周”再补，而是每次高影响改动完成后就顺手更新一条。

因为这时候最清楚：

- 差异是什么
- 为什么这么改
- 后续谁应该优先被参考

### 差异记录不是用来否定 base 的

要记住：

- base 仍然是参考基线
- 但长期维护项目首先是“当前项目”
- 如果当前项目已经偏离 base，后续维护优先尊重当前项目现状和业务原因

这条原则对 AI 尤其重要。

---

## 九、一句话总结

差异记录真正解决的是：

**让人和 AI 在长期维护项目里，知道什么时候该参考 base，什么时候必须尊重当前项目。**

只要这件事不清楚，项目越大，误改概率就越高。

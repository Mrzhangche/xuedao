# 示例：新增一个普通功能怎么走

这份文档给初级程序员直接看。

目标不是讲抽象方法，而是用一个**真实工作里很常见的普通功能**，带你走一遍：

**接到需求 → 判断级别 → 判断涉及哪些端 → 先让 AI 做什么 → 产出哪些文档 → 按什么顺序开发 → 去哪里找同类代码 → 开发完怎么自测和交付。**

---

## 一、先看这次示例是什么

这次我们用一个很常见、也比较真实的例子：

**新增“优惠券管理”功能。**

场景设定：

- 运营希望在后台创建优惠券
- 可以设置优惠券名称、面额、使用门槛、有效期、状态
- 后端要提供优惠券的增删改查接口
- 本次先做 **后台 + 后端**
- 先不做用户端领取，也不做复杂发券逻辑

这是一个非常典型的“普通功能”：

- 不只是改个文案
- 也不是系统级重构
- 有后台页面
- 有后端接口
- 可能还有数据库表
- 但整体范围可控

这类功能，最适合初级程序员拿来练“真实工作流程”。

---

## 二、需求长什么样

真实工作里，你接到的需求通常不会很规范，可能长这样：

> 帮后台加一个优惠券管理功能。
> 
> 运营可以新增、编辑、停用优惠券。
> 
> 优惠券要有名称、减免金额、最低使用金额、开始时间、结束时间、状态。
> 
> 后台需要列表查询。
> 
> 后端要把增删改查接口补上。

有时候产品再补一句：

> 这期先做管理功能，不做领取和核销。

你拿到这种需求后，**第一反应不要是“我先建表”**。

你先要做的是：**把需求翻译成一个可执行任务。**

翻译后应该变成：

- 要做一个“优惠券管理”模块
- 后台需要：列表页、搜索、新增、编辑、启停用
- 后端需要：优惠券 CRUD 接口
- 数据层可能需要：优惠券表
- 本次不做：领取、发放、核销、用户端展示

只要你能把任务翻译成这几条，你就已经比“直接开写”的人更稳了。

---

## 三、第一步怎么判断这是 L2

先回忆一下最常用的分级思路：

- **L1**：小修复、小改动
- **L2**：普通功能、单模块功能
- **L3**：复杂功能、多角色流程
- **L4**：系统级、跨多个核心端的高影响任务

这次为什么是 **L2**？

因为它符合 L2 的典型特征：

1. **新增的是一个普通业务功能，不是小修复**
   - 不是改文案
   - 不是改一个判断
   - 不是修一个已有 bug

2. **范围是一个模块内可控的新增**
   - 主要就是“优惠券管理”这个模块
   - 不会一下子牵出整套订单主流程重做

3. **通常会影响多个文件，但还没到系统级改造**
   - 后台会有页面、表单、接口调用
   - 后端会有 controller / service / entity 之类的改动
   - 可能加一张表

4. **需要先明确字段和接口，但不需要很重的方案设计**
   - 名称、面额、门槛、有效期、状态这些字段要先定
   - 列表查询条件、保存参数要先定
   - 但不需要写一大套系统架构设计

5. **这次是“后端 + 后台”的普通联动**
   - 会跨两个端
   - 但仍然是一个普通业务模块，不属于 L4 那种核心系统改造

### 一个给新人的简单判断法

如果一个需求满足下面大多数条件，你大概率就可以先按 L2 走：

- 不是小改动
- 是新增一个普通模块或普通页面
- 需要前后端配合，但范围可控
- 需要字段、接口、表单、列表这些常规开发内容
- 不涉及登录、权限体系、订单主流程、支付主流程这种核心链路

### 这一步最后要得出的结论

你要敢写出一句判断：

> 这是一个 L2 普通功能，涉及后台和后端两个端，建议先让 AI 帮我整理需求说明、技术设计和实施步骤，再开始开发。

这句话很重要，因为它会直接决定你后面怎么推进。

## 四、怎么判断涉及哪些端

这一步也非常关键。

你要做的不是“看起来像前端需求，所以先写页面”，而是先拆：

### 1. 哪些内容一定是后端的

这次需求里，下面这些一看就知道是后端：

- 优惠券列表接口
- 新增优惠券接口
- 编辑优惠券接口
- 删除/停用/启用接口
- 优惠券数据存储
- 查询条件处理
- 数据校验（例如结束时间不能早于开始时间）

所以：**后端一定涉及。**

### 2. 哪些内容一定是后台的

这次需求里，下面这些一看就知道是后台：

- 优惠券管理列表页
- 搜索条件
- 新增弹窗或新增页面
- 编辑弹窗或编辑页面
- 状态显示
- 操作按钮（编辑、停用、启用等）

所以：**后台一定涉及。**

### 3. uni 端这次要不要动

需求已经说了：

- 这期先做管理功能
- 不做领取和核销
- 不做用户端展示

所以这次：**uni 不涉及。**

### 4. 这类需求最终怎么判断

最终你应该得出：

- **涉及后端**：因为有接口和数据
- **涉及后台**：因为有管理页面
- **不涉及 uni**：因为没有用户端页面和用户操作流程
- **这不是三端联动，只是后台 + 后端的普通功能**

### 新人最容易犯的错

- 只看到“新增页面”，没想到后端接口也得补
- 先写后台，写到一半才发现接口字段没定
- 没问清楚这期是否包含 uni，结果多做了

所以你拿到需求后，要先做一句话判断：

> 这是后端 + 后台任务，不涉及 uni；应先把字段、接口和页面结构定下来，再进入开发。

## 五、应该先让 AI 做什么

对于这种 L2 任务，**最好的第一步，不是让 AI 直接开写代码**。

更稳的做法是先让 AI 帮你完成下面几件事：

### 1. 帮你判断任务级别和涉及端

先让 AI 帮你确认：

- 这是 L2 吗
- 涉及哪些端
- 哪些是本期范围，哪些不是

这样做的好处是：

- 你不容易一上来就做多
- 你能先把边界划清楚
- 后面生成文档时，范围更稳定

### 2. 帮你拆需求范围

让 AI 先把需求拆成：

- 页面要做什么
- 接口要做什么
- 数据字段有哪些
- 状态有哪些
- 本期不做什么

这一步特别有用，因为初级程序员最容易漏的就是“本期不做什么”。

### 3. 帮你先找同类代码

这是非常重要的一步。

你应该先让 AI 去仓库里找：

- 有没有类似的后台管理模块
- 有没有类似的增删改查页面
- 有没有类似的后端 CRUD 模块
- 有没有类似的状态字段、时间区间、启停用逻辑

因为真实工作里，**大部分普通功能都不是从零发明，而是照着同类模块做。**

### 4. 帮你先产出简版开发文档

先让 AI 生成：

- 需求说明
- 技术设计
- 开发实施步骤

如果你直接跳过这一步，后面很容易出现：

- 页面先写了，但接口字段不一致
- 后端字段定义了，后台表单没跟上
- 本来只做启停用，结果顺手写成了删除 + 发券 + 核销

## 六、AI 会先产出哪些文档

对于这个“优惠券管理”L2 例子，建议 AI 先产出下面这些文档。

不一定每个项目都叫这个名字，但内容大致应该有。

### 1. 需求说明

作用：把“要做什么、不做什么”说清楚。

建议至少包含：

- 功能目标
- 使用角色（运营/管理员）
- 本期范围
- 非本期范围
- 页面功能点
- 接口功能点
- 关键字段

这份文档写完后，你应该能回答：

- 我这期到底交付什么
- 哪些内容不要顺手做进去

### 2. 技术设计

作用：把“代码大概要怎么落”说清楚。

建议至少包含：

- 涉及哪些端
- 后端模块落点
- 后台页面落点
- 数据表或字段设计
- 接口设计
- 状态设计
- 参数校验规则

这份文档写完后，你应该能回答：

- 先写哪边
- 每边大概改哪些文件
- 字段和接口有没有定清楚

### 3. 开发实施文档

作用：把“开发顺序”说清楚。

建议至少包含：

- 第一步做什么
- 第二步做什么
- 每一步完成标准是什么
- 联调时看什么
- 自测时看什么

这份文档最适合初级程序员，因为它直接解决“我现在下一步该干嘛”。

### 4. 如果有数据库变动，再补数据库说明

如果这次要新建优惠券表，建议补：

- 表名
- 字段
- 字段含义
- 是否允许为空
- 默认值
- 索引建议

### 5. 如果接口比较多，再补 API 说明

如果这次接口比较清楚，也建议补：

- 列表接口
- 详情接口
- 新增接口
- 编辑接口
- 启停用接口
- 请求参数、返回字段、错误提示

### 初级程序员可以这样理解

L2 的文档不是为了“写得像大公司 PPT”。

而是为了防止三种典型返工：

- 范围返工
- 字段返工
- 开发顺序返工

## 七、接下来开发顺序是什么

这一段最接近真实工作。

对于这个例子，推荐顺序如下：

### 第 1 步：先补简版文档，定范围

先定清楚：

- 本期只做后台 + 后端
- 本期不做用户领取和核销
- 字段有哪些
- 状态有哪些
- 列表、保存、启停用这几个动作是否足够

这一步完成后，你再开始看代码。

### 第 2 步：先找同类模块

先不要急着新建文件。

你先在仓库里找一个**最像“优惠券管理”的普通后台模块**，比如：

- 活动管理
- 分类管理
- 公告管理
- 标签管理
- 会员等级管理
- 配置项管理

找的时候重点看：

- 后台列表页怎么写
- 新增/编辑表单怎么写
- 状态字段怎么展示
- 启停用操作怎么写
- 后端 CRUD 是怎么组织的
- 分页、查询条件、保存参数怎么写

**先找同类，再仿照写。**

这是普通功能开发里最重要的习惯之一。

### 第 3 步：先做后端基础能力

推荐先做后端，因为后台页面最终要依赖接口和字段。

通常顺序是：

1. 明确数据结构
2. 建表或补 entity/model
3. 写列表接口
4. 写新增接口
5. 写编辑接口
6. 写启停用/删除接口
7. 补基本参数校验
8. 先用接口自测一轮

为什么先做后端？

因为后台页面很多内容都依赖这些信息：

- 字段名
- 状态枚举
- 查询参数
- 返回结构

如果后端没定，后台很容易一边写一边改。

### 第 4 步：再做后台页面

后端定下来后，再做后台会顺很多。

后台通常按这个顺序：

1. 接接口定义
2. 搭列表页
3. 搭搜索区
4. 搭新增表单
5. 搭编辑表单
6. 接状态展示和状态操作
7. 跑通新增、编辑、查询、启停用

### 第 5 步：联调

联调时重点看：

- 字段名是否一致
- 时间格式是否一致
- 状态值是否一致
- 列表查询条件是否生效
- 新增和编辑回显是否正常
- 停用后列表展示是否正确

### 第 6 步：补测试和交付说明

不要觉得“代码能跑就结束了”。

真实工作里，最后这一步非常重要。

你至少要补：

- 我做了哪些内容
- 哪些没做
- 我怎么测的
- 需要别人怎么验收
- 有没有已知限制

## 八、开发时去哪里找同类代码

这一步非常实用。

### 1. 先找“业务上相似”的模块

优先找和你这次功能形态接近的模块，而不是只看名字像不像。

比如“优惠券管理”更像：

- 有列表
- 有搜索
- 有新增/编辑
- 有状态
- 有时间范围

那你就应该找同样具备这些特点的模块。

### 2. 在后台端重点找这些内容

你要重点看同类模块的：

- 列表页面
- 表单组件
- 搜索区写法
- 状态列写法
- 操作列写法
- API 调用文件
- 页面初始化和保存逻辑

### 3. 在后端重点找这些内容

你要重点看同类模块的：

- controller
- service
- entity / model
- DTO / 参数定义
- 分页查询
- 保存逻辑
- 状态更新逻辑

### 4. 先找“最普通的 CRUD 模块”

如果你一上来找一个特别复杂的业务模块，你会被绕晕。

对初级程序员来说，最佳做法是：

**先找一个结构最标准、最简单、最像模板的 CRUD 模块。**

### 5. 让 AI 帮你做这件事

你可以直接让 AI：

- 在仓库里找最像“优惠券管理”的现有模块
- 列出后台对应文件
- 列出后端对应文件
- 告诉你哪些文件适合参考
- 按“列表 / 表单 / 接口 / 状态”分类说明

这样你会比自己瞎搜快很多。

## 九、一个真实可执行的开发顺序示例

下面给你一个“新人照着走就行”的版本。

### 第 0 步：把需求和范围写清楚

先写出：

- 功能：新增优惠券管理
- 范围：后台 + 后端
- 本期不做：领取、发放、核销、用户端展示
- 核心字段：名称、面额、最低使用金额、开始时间、结束时间、状态

### 第 1 步：让 AI 判断任务并拆步骤

目标：不要直接开工，先把路走顺。

### 第 2 步：让 AI 找同类代码

目标：找到最像的后台模块和后端模块。

### 第 3 步：让 AI 先生成简版文档

至少要有：

- 需求说明
- 技术设计
- 开发实施

### 第 4 步：先做后端

你去落：

- 数据表 / entity
- 列表接口
- 新增接口
- 编辑接口
- 启停用接口

并先自己调通接口。

### 第 5 步：再做后台

你去落：

- 列表页
- 搜索区
- 新增表单
- 编辑表单
- 状态展示与操作

### 第 6 步：联调

把后台和后端一起跑通。

### 第 7 步：自测

按“新增、编辑、查询、启停用、边界值”完整测一遍。

### 第 8 步：补交付说明

告诉别人：

- 我做了什么
- 我怎么测的
- 你们怎么验
- 哪些暂时没做

## 十、开发完后怎么自测

自测不是“我点了一下能打开页面”。

对这个例子，至少应该覆盖下面这些。

### 1. 基础流程自测

- 能打开优惠券管理页面
- 能看到列表
- 能新增一条优惠券
- 新增后列表能查到
- 能编辑已存在优惠券
- 编辑后列表展示正确
- 能停用优惠券
- 能重新启用优惠券

### 2. 查询自测

- 按名称搜索是否生效
- 按状态搜索是否生效
- 按时间范围搜索是否生效（如果有）
- 分页是否正常

### 3. 表单自测

- 必填项是否校验
- 面额是否允许非法值
- 最低使用金额是否允许非法值
- 开始时间和结束时间顺序是否正确
- 编辑时回显是否正确

### 4. 状态流转自测

- 启用状态显示是否正确
- 停用后是否能看到正确状态
- 已停用数据能否被再次编辑，是否符合需求

### 5. 边界情况自测

- 名称过长怎么办
- 金额为 0 是否允许
- 结束时间早于开始时间是否拦截
- 查询条件为空时是否正常返回列表

### 6. 联调自测

- 后台传给后端的字段名是否一致
- 后端返回给后台的字段是否够用
- 时间格式在页面展示是否正常
- 状态值和文案是否对得上

### 一个适合新人的原则

自测时，不要只测“正常新增成功”。

至少要同时测：

- 一个正常路径
- 一个编辑路径
- 一个状态切换路径
- 一个校验失败路径

这样你的自测才比较像真实工作。

## 十一、开发完后怎么补交付说明

很多新人会漏这一步。

实际上，交付说明就是告诉测试、产品、同事：

**这次改了什么，你们该怎么验。**

一份够用的交付说明，建议至少写下面这些：

### 1. 本次交付内容

例如：

- 后台新增优惠券管理页面
- 后端新增优惠券增删改查接口
- 支持名称、面额、门槛、有效期、状态管理
- 支持列表查询和启停用

### 2. 本次不包含内容

例如：

- 不包含用户领取
- 不包含优惠券核销
- 不包含用户端展示
- 不包含发券逻辑

### 3. 自测结果

例如：

- 已验证新增、编辑、列表查询、启停用
- 已验证必填和时间范围校验
- 已验证编辑回显正常

### 4. 验收建议

例如：

- 先新增一条优惠券
- 再按名称搜索
- 再编辑金额和时间
- 再停用 / 启用观察状态变化

### 5. 如果有已知限制，也写出来

例如：

- 当前只支持后台管理，不支持用户端领取
- 当前未接入自动发券

### 一个很真实的提醒

写清“本次不做什么”，可以减少很多无效来回。

## 十二、可以直接复制给 AI 的示例提示词

下面这些提示词，都是给初级程序员直接复制用的。

### 1. 先判断任务级别和涉及端

```text
我接到一个普通功能，请先不要直接写代码，先帮我判断任务级别和影响范围。

需求如下：
新增优惠券管理功能：
- 运营可以在后台新增、编辑、停用优惠券
- 优惠券包含名称、减免金额、最低使用金额、开始时间、结束时间、状态
- 后端需要提供优惠券增删改查接口
- 后台需要列表、搜索、新增、编辑、启停用
- 这期先不做用户领取、发券、核销、用户端展示

请先输出：
1. 这是 L1 / L2 / L3 / L4 哪一级，为什么
2. 涉及后端 / 后台 / uni 中哪些端，为什么
3. 本期范围是什么
4. 非本期范围是什么
5. 推荐开发顺序是什么
```

### 2. 让 AI 先找同类代码

```text
请先不要写新代码，先在当前仓库里帮我找“最像优惠券管理”的同类实现。

我要做的功能是：
后台 + 后端的优惠券管理，包含列表、搜索、新增、编辑、状态启停用。

请你帮我输出：
1. 后台最值得参考的 2~3 个模块
2. 后端最值得参考的 2~3 个模块
3. 每个模块分别适合参考什么（列表、表单、状态、CRUD、分页、查询条件等）
4. 如果我照着现有结构开发，建议我优先看哪些文件
```

### 3. 让 AI 先产出文档

```text
这是一个 L2 普通功能，请先不要直接开发，先按当前项目基座帮我整理文档。

需求如下：
新增优惠券管理功能：
- 后台支持列表、搜索、新增、编辑、启停用
- 后端支持优惠券增删改查
- 字段包括名称、减免金额、最低使用金额、开始时间、结束时间、状态
- 本期不做领取、发券、核销、用户端展示

请先生成：
1. 需求说明
2. 技术设计
3. 开发实施文档
4. 如果需要，再补数据库说明和 API 说明

要求：
- 全部中文
- 适合初级程序员阅读
- 每一步写清楚先做什么、后做什么
- 明确本期范围和非本期范围
```

### 4. 让 AI 按顺序带你开发

```text
请按“初级程序员可以直接照着做”的方式，带我完成这个 L2 功能。

功能是：新增优惠券管理（后台 + 后端）。
本期范围：列表、搜索、新增、编辑、启停用。
不做：领取、发券、核销、用户端展示。

要求：
1. 先找同类代码
2. 先定字段和接口
3. 先做后端，再做后台
4. 每一步告诉我应该改哪些文件
5. 每一步告诉我改完怎么自测
6. 不要顺手扩功能
```

### 5. 开发完后让 AI 帮你补交付说明

```text
我已经完成“优惠券管理”功能开发，请根据下面信息帮我整理一份交付说明。

功能范围：
- 后台：列表、搜索、新增、编辑、启停用
- 后端：优惠券增删改查
- 字段：名称、减免金额、最低使用金额、开始时间、结束时间、状态

本期不包含：
- 领取
- 发券
- 核销
- 用户端展示

请输出：
1. 本次交付内容
2. 本次不包含内容
3. 自测说明
4. 验收建议
5. 已知限制
```

## 十三、一句话总结这类普通功能该怎么走

如果你是初级程序员，接到这种“后台 + 后端”的普通功能，最稳的走法就是：

**先判断它是 L2 → 再判断涉及哪些端 → 先让 AI 帮你定范围、找同类、出简版文档 → 先做后端 → 再做后台 → 联调 → 自测 → 补交付说明。**

不要一上来就写代码。

你真正要学会的，不只是把代码写出来，而是学会一条**真实工作里可以稳定复用的开发路径**。

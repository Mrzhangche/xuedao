# 阶段化开发怎么协作

这份说明是给团队成员看的。

以后默认和 AI 协作开发，不是“一句话丢给 AI，等它一次性做完”，而是走**阶段化执行**。

## 默认怎么走

默认按下面 6 个阶段推进：

1. 需求分析与任务分类
2. 开发前文档产出
3. 文档审核与迭代
4. 分步骤开发
5. 每步验收与纠偏
6. 收尾补齐文档

在这个流程里，可以把**任务执行卡**理解为“阶段化协作的执行记录”。

- `06` 这篇文档，讲的是**协作方式怎么走**
- `07` 这篇文档，给的是**任务执行卡怎么记、怎么用**

通常建议在任务已经判断为需要分阶段推进、多人协作、后续可能回看时，就先把任务执行卡建起来。

后面每次阶段推进、人工确认、发现偏差要回改时，都要顺手同步更新执行卡。这样做的目的不是重复写文档，而是让当前做到哪一步、为什么停、哪里改过，一眼就能看清，避免团队只知道流程，不知道这次任务现在走到哪里。

简单理解：

- 先把事情说清楚
- 再把文档对齐
- 再一小步一小步开发
- 每一步都能人工检查
- 发现理解错了，先改文档，再继续写代码

## 什么时候需要人工确认

默认下面几个地方需要人工确认：

- 任务分级和影响范围判断完之后
- 开发前文档产出之后
- 文档每次大改之后
- 每一个开发步骤完成之后
- 最后测试、交付、文档补齐之后

没确认前，AI 不应该直接跳到下一阶段。

## 如果中途改需求怎么办

如果中途发现需求变了，或者某一步理解错了，不是直接在代码上硬改过去。

正确做法是：

1. 先回改相关文档
2. 再确认这次变更会影响哪些后续步骤
3. 再继续开发

通常要回改这些内容：

- 需求说明
- 技术设计
- 开发实施步骤
- 接口说明
- 数据库说明（如果有）
- 测试和交付说明

## 为什么这样更稳

因为大多数返工都不是“不会写代码”，而是：

- 一开始理解就偏了
- 文档和代码越做越不一致
- 跨端任务没有先把接口和字段定清楚
- 做完一大坨以后才发现方向不对

阶段化协作的好处是：

- 更容易让团队一起审需求
- 更容易在开发前发现理解偏差
- 更容易把大任务拆成可检查的小步骤
- 更容易在中途纠偏，而不是最后返工
- 更适合长期维护项目

## 一句话记住

以后默认是：**先对齐文档，再分步开发；每步做完先验收，有偏差先回文档。**

# 任务执行卡模板

这份文档是给团队成员和 AI 一起用的。

目的不是写一堆形式化材料，而是给每个需求配一张**统一的执行卡**，让大家随时知道：

- 当前做到哪个阶段了
- 已经确认了什么
- 还有哪些点没确认
- 下一步该做什么
- 如果中途推翻，要回改哪些文档

这样做的好处是：

- AI 不会一口气把任务做穿，忘了等确认
- 团队成员接手时，能快速看懂任务推进到哪
- 文档、代码、当前状态能保持同步
- 中途改需求时，不会只改代码、不改文档

## 这张执行卡是干什么的

任务执行卡就是一张“当前任务状态总表”。

它主要解决 4 个问题：

1. 记录当前任务在哪个阶段
2. 记录这一阶段已经得出的结论
3. 记录还有哪些问题要等用户或团队确认
4. 记录下一步动作，以及如果推翻要回改什么

你可以把它理解成：

- 不是需求文档
- 不是技术设计文档
- 不是测试报告
- 而是把这些文档和当前推进状态串起来的“总控卡”

## 什么时候创建

默认建议这样做：

- **L1 小修复**：可选；如果任务很小、很明确，可以不单独建卡
- **L2 普通功能**：建议在阶段 1 结束后立刻创建
- **L3 复杂功能 / 多人协作任务**：必须创建
- **L4 系统级 / 跨端 / 高影响任务**：必须创建

更直接一点：

- 只要任务需要走“阶段化开发”，就应该创建任务执行卡
- 最晚不要晚于“开发前文档产出阶段”
- 最好在“需求分析与任务分类阶段”确认后就建好

## 默认要记录哪些字段

一张可用的任务执行卡，至少要记录下面这些字段：

- 任务名称
- 当前阶段
- 任务等级（L1/L2/L3/L4）
- 涉及端（vue / midway / uni / 跨端）
- 需求摘要
- 当前结论
- 已完成内容
- 待确认问题
- 下一步动作
- 相关文档清单
- 相关代码目录/模块
- 如果被推翻，需要回改哪些文档
- 当前状态（待确认 / 进行中 / 已完成 / 已暂停）

如果团队需要，也可以再补：

- 负责人
- 创建时间
- 最近更新时间
- 当前阻塞点
- 本轮确认人

但最少不要少于上面那一组核心字段。

## 推荐怎么维护

推荐按下面规则维护，最不容易乱：

1. **阶段 1 结束后创建执行卡**
2. **进入下一阶段前，先更新执行卡**
3. **用户确认后，立刻更新“当前阶段 / 当前结论 / 当前状态 / 下一步动作”**
4. **如果某一步被推翻，先更新执行卡，再回改相关文档和代码**
5. **开发结束时，不删除执行卡，保留完整推进记录**

简单说：

- 文档有变化，要更新执行卡
- 阶段有变化，要更新执行卡
- 结论有变化，要更新执行卡
- 回滚或推翻，也要更新执行卡

## 字段怎么填，最实用

### 1）任务名称
不要只写“改一下功能”。

推荐写法：

- 新增优惠券批量停用功能
- 补齐订单详情页发票展示
- 调整用户端登录失效后的跳转流程

### 2）当前阶段
直接写当前卡在哪一阶段，不要模糊写“处理中”。

例如：

- 阶段 1：需求分析与任务分类
- 阶段 2：开发前文档产出
- 阶段 3：文档审核与迭代
- 阶段 4：分步骤开发
- 阶段 5：每步验收与纠偏
- 阶段 6：收尾补齐文档

### 3）当前状态
推荐只用这 4 个值，团队最容易统一：

- 待确认：当前阶段产物已经出来，正在等人确认
- 进行中：当前阶段正在推进
- 已完成：整个任务已结束
- 已暂停：当前任务先停住，等外部条件

### 4）当前结论
这里不要写过程废话，直接写现在已经成立的结论。

比如：

- 已确认本次属于 L2 普通功能，只涉及 vue 端和 midway 端
- 已确认需要补需求、技术、实施、API 四份文档
- 已确认接口字段名以当前项目已有命名为准，不按 base 新开一套

### 5）待确认问题
这里只写“没确认就不能继续推进”的问题，不要把普通备注都塞进来。

比如：

- 是否保留旧页面入口
- 审批状态字段是复用现有枚举，还是新增枚举
- 本次是否需要同时支持 uni 端展示

### 6）如果被推翻，需要回改哪些文档
这一栏非常重要，不能省。

推荐按“变更类型 → 回改文档”来写：

- 需求变了：回改需求说明、产品设计、开发实施
- 接口变了：回改技术设计、API 文档、数据库文档（如有）
- 步骤变了：回改开发实施、测试检查项、交付说明

这样 AI 或团队成员中途接手时，不会只知道“要改”，却不知道“先改哪几份文档”。

## 一个很短的小示例

```md
# 任务执行卡

## 基本信息

- 任务名称：新增优惠券批量停用功能
- 当前阶段：阶段 2
- 当前状态：待确认
- 任务等级：L2
- 涉及端：vue / midway
- 负责人：AI + 当前开发同学
- 创建时间：2026-04-15
- 最近更新时间：2026-04-15

## 需求摘要

- 本次要在优惠券管理页支持批量停用
- 目标是让运营能一次停用多张未失效优惠券
- 影响范围包括后台页面、后端接口、优惠券状态校验

## 当前结论

- 已确认本次是 L2 普通功能
- 已确认涉及 vue 和 midway，不涉及 uni
- 已产出需求说明、技术设计、开发实施文档
- 当前停在阶段 2，等待用户确认文档后进入开发

## 已完成内容

- [x] 完成任务分级
- [x] 完成影响范围判断
- [x] 完成开发前文档初稿

## 待确认问题

- [ ] 批量停用是否允许操作“未开始”状态的优惠券
- [ ] 页面上是否需要二次确认弹窗

## 下一步动作

- 下一步准备：根据确认结果进入阶段 4，先做后端批量停用接口
- 进入条件：用户确认当前文档
- 完成后要同步更新：执行卡、开发实施文档

## 相关文档清单

- `docs/01-modules/coupon/01-requirements.md`
- `docs/01-modules/coupon/03-technical-design.md`
- `docs/01-modules/coupon/06-implementation.md`

## 相关代码目录 / 模块

- `cool-admin-vue/src/modules/coupon/`
- `cool-admin-midway/src/modules/coupon/`

## 如果被推翻，需要回改哪些文档

- 需求被推翻时回改：需求说明、开发实施文档
- 技术方案被推翻时回改：技术设计、开发实施文档
- 接口 / 字段被推翻时回改：技术设计、API 文档
- 步骤顺序被推翻时回改：开发实施文档、测试检查项
```

## 建议放哪里

如果是项目里的真实任务，建议放在对应模块文档目录里，方便和需求、设计、实施文档放在一起。

例如：

```text
docs/
  01-modules/
    coupon/
      00-task-card.md
      01-requirements.md
      03-technical-design.md
      06-implementation.md
```

如果当前任务还没进入模块目录，也可以先临时放在：

- `docs/tasks/任务名.md`
- 或模块目录下的 `00-task-card.md`

核心原则只有一个：**执行卡要和这次任务的文档放在一起，后面的人能一眼找到。**

## 一句话记住

任务执行卡不是额外负担，而是把“当前阶段、确认点、下一步、回改范围”固定下来，让阶段化开发真的能闭环。

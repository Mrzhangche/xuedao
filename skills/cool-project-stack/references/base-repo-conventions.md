# Base 仓库团队约定

这份文件用于补充 `base` 仓库内的通用约定，主要给总控 skill 和 `cool-uni` 兜底使用。

## 适用原则

- `base` 是新项目初始化基线。
- 当前项目如果已经存在实现，优先尊重当前项目，不强行改回模板。
- 当前项目缺少某类实现时，再回看 `base` 对应模板。

## 推荐开发顺序

跨端任务默认按以下顺序推进：

1. `cool-admin-midway`
2. `cool-uni`
3. `cool-admin-vue`

原因：

- 服务端接口和数据结构要先稳定
- 移动端通常依赖服务端接口
- 后台端更多是管理、运营、配置与验证入口

## 初始化阶段约定

开始写业务代码前，先确认：

- 项目名
- 接口域名或代理
- 数据库连接
- 登录方式
- 页面入口
- 需要的模块范围

不要边写边猜配置。

## 文档约定

新功能开发默认先出文档，再开始编码。

中等及以上任务默认输出：

1. 需求说明
2. 技术设计
3. 数据库设计
4. 接口或数据结构说明
5. 开发实施说明
6. 联调与测试清单
7. 交付说明

按场景裁剪：

- 不涉及数据库变更时，可以不单独输出数据库设计文档
- 不涉及接口变化时，可以把接口说明合并到技术设计里
- 但新功能至少要有：需求说明、技术设计、开发实施说明

文档要写给团队成员和初级程序员看，不写空泛概念。

## 命名约定

- 统一命名规则优先参考 `./naming-conventions.md`
- 当前项目已有命名优先于 `base`
- 新增模块、字段、接口、页面、变量时，不要临时发明另一套名字

## Vue 侧补充

- 后台页面优先复用 `cl-crud`、`cl-table`、`cl-upsert` 等现有模式。
- 不轻易改核心包和公共底座。
- 模块优先放 `src/modules`，公共能力再考虑 `src/plugins`。

## Midway 侧补充

- 严格模块化开发。
- 标准 CRUD 尽量走框架能力，不重复造接口。
- 普通业务优先 TypeORM API，不随意写原生 SQL。
- 多模块解耦优先事件，不把所有逻辑塞进一个 Service。

## Uni 侧补充

- UI 优先使用 `cool-ui`。
- 页面优先沿用 `pages`、`router`、`config`、`cool` 现有结构。
- 先确认页面入口、登录态、接口接入方式，再写具体页面。

# 文档目录整理建议

## 目标

让项目文档满足下面 4 个目标：

- 开发前有文档，不靠口头约定
- 开发后能回写，不让文档和代码脱节
- 新人能快速找到“系统怎么组成、功能怎么用、这次改了什么”
- 不把所有内容挤进一个总文档里

## 文档根目录规则

- 当前维护的是模板仓库 `base`，默认输出到 `base/docs`
- 当前维护的是具体业务项目，默认输出到项目根目录 `docs`
- 除非用户明确指定其他位置，否则都按上面的默认规则执行

## 先看这一条：任务分级决定文档多少

### L1：小修复

常见情况：

- 改文案、样式、提示语
- 修一个明确的小 bug
- 调整一个小判断、小字段、小接口处理
- 不新增模块，不改主流程，不改数据库

文档要求：

- 不要求全套文档
- 至少补简短改动说明
- 提交前做最小自测

### L2：普通功能

常见情况：

- 新增一个普通页面、表单、列表
- 在一个模块内新增一段业务逻辑
- 新增或调整接口，但范围可控

文档要求：

- 开发前至少补：需求、技术、实施
- 涉及接口变更：补 API 文档
- 涉及数据库变更：补数据库文档

### L3：复杂功能 / 多人协作任务

常见情况：

- 新模块、新流程、跨角色流程
- 一个需求会影响多个目录、多个页面、多个接口
- 多人协作、分阶段开发、分批交付

文档要求：

- 需要较完整文档链路
- 至少包括：需求、产品、技术、实施、测试、交付
- 涉及接口必须补 API 文档
- 涉及数据库必须补数据库文档

### L4：系统级 / 跨端 / 高影响任务

常见情况：

- 跨端任务
- 登录、权限、核心流程、主数据调整
- 系统结构调整
- 同时影响后端、后台、Uni 中两个及以上端

文档要求：

- 必须补全套文档
- 必须写清楚各端怎么配合
- 只要有接口变化，必须补 API 文档
- 只要有数据结构或表结构变化，必须补数据库文档
- 交付前必须补联调和交付说明

## 推荐目录结构

```text
docs/
  README.md
  00-system/
    system-overview.md
    system-function-manual.md
  01-modules/
    student/
      01-requirements.md
      02-product-design.md
      03-technical-design.md
      04-database.md
      05-api.md
      06-implementation.md
      07-test-checklist.md
      08-delivery.md
```

## 为什么这样分

### 根目录 `README.md`

这是给人看的总入口，不负责写模块细节，只负责回答下面几个问题：

- 这个系统是什么
- 先看哪几份文档
- 文档目录怎么分
- 主要模块有哪些
- 新人想快速了解全貌时，推荐阅读顺序是什么

如果没有这份文档，系统文档即使很全，也不容易一眼看懂。

### 1. `00-system`

放长期维护、跨模块复用的项目级文档：

- `system-overview.md`
  写系统目标、端划分、模块边界、技术栈、权限角色、部署与外部依赖
- `system-function-manual.md`
  写已经交付的功能入口、适用角色、主要流程、注意事项、限制条件

这两个文件不是一次性文档，而是每次功能完成后持续更新。

### 2. `01-modules/模块名`

每个模块单独一个目录，便于把需求、设计、数据库、接口、测试、交付串起来。

推荐一个模块目录里按固定顺序放：

- `01-requirements.md`
- `02-product-design.md`
- `03-technical-design.md`
- `04-database.md`
- `05-api.md`
- `06-implementation.md`
- `07-test-checklist.md`
- `08-delivery.md`

不是每次都必须 8 份全开，但文件顺序尽量固定。

## 命名建议

- 目录名优先使用模块英文代码名，如 `student`、`course-schedule`
- 文档标题可以用中文
- 不建议目录名直接使用“学生模块最终版”“学生模块最新”这类会不断失效的名字
- 同一个模块长期维护，优先在原目录内迭代，不反复新建同名目录

## 什么时候更新哪些文档

### 开发前

#### L1

- 可以不建完整模块目录
- 至少补简短改动说明，或者在任务记录里写清楚改了什么

#### L2

至少准备：

- `README.md`（如果项目文档入口还没有）
- `01-requirements.md`
- `03-technical-design.md`
- `06-implementation.md`

按场景追加：

- 有前后端联调或新接口，补 `05-api.md`
- 有表结构变化，补 `04-database.md`

#### L3 / L4

至少准备：

- `README.md`
- `01-requirements.md`
- `02-product-design.md`
- `03-technical-design.md`
- `06-implementation.md`

必须按场景追加：

- 有表结构变化，补 `04-database.md`
- 有前后端联调或新接口，补 `05-api.md`

### 开发完成后

#### L1

至少更新：

- 改动说明
- 必要的自测结果

#### L2 及以上

至少更新：

- `07-test-checklist.md`
- `08-delivery.md`
- `README.md`
- `00-system/system-function-manual.md`

如果影响系统结构、角色权限、模块边界、页面入口、外部依赖、部署配置，再继续更新：

- `00-system/system-overview.md`

## 维护原则

- 能更新旧文档就不要重开一份新总文档
- 模块文档写当前模块，系统文档写跨模块共识
- `README.md` 只做导航和阅读指引，不承载模块细节
- 文档里必须写清楚文件落点、页面入口、接口路径、关键字段、验收方式
- 不要保留明显已经失效的旧描述
